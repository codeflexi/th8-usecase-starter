TH8

Procurement Contract · UX Flow Comparison

Current (SAP/Workflow/OCR/Dashboard) vs TH8.AI Overlay

Scope: Purchase Contract Only TH8 Decision Center

Executive Summary

TH8 ไม่ได้ “แทน SAP” แต่เป็นชั้น Overlay ที่ทำให้ “Pending เกิดจากหลักฐานจริง” และ “ตัดสินใจได้เร็วขึ้น”

ระบบเดิมทำได้ดีเรื่องบันทึกธุรกรรม, ลำดับอนุมัติ, และรายงานภาพรวม แต่จุดที่ยังเป็นคอขวดในโลกจริงคือ การเทียบสัญญา + การอธิบายเหตุผล + การทำ audit ที่ย้อนกลับไปหาเอกสารได้ทันที. TH8 เข้าไปเสริม “จุดตัดสินใจ” ให้เป็นมาตรฐานและตรวจสอบได้ โดยไม่ต้องรื้อระบบเดิม

What changes

  • ก่อน

    อนุมัติบน “หน้าจอ workflow” แต่หลักฐาน/สัญญาอยู่คนละที่

  • หลังมี TH8

    อนุมัติบน “decision card” ที่มี delta + evidence link + reason (บังคับ)

Legend

Node = ขั้นตอนงาน Data = ข้อมูลที่เกิดขึ้น TH8 Overlay Pending ที่ “ควรดูจริง” Auto-pass เคสปกติ

Scope Guardrails

  • โฟกัสเฉพาะ “สัญญาซื้อ” (Contract-based)
  • ไม่เพิ่ม use case ใหม่
  • ไม่ขยายไปฝั่งขาย / รายได้
  • ทำ 1 flow ให้สมบูรณ์ที่สุด

C) Policy Engine (ทำให้ “Pending เกิดจริง”)

นโยบายอ่านแบบธุรกิจ: rule_id + severity + delta + pending_reason

Phase 1

Phase 3

ตัวอย่าง KPI

เก็บ event + policy_version/hash + evidence + decision reason

Phase 4

Audit-ready =

  • ชี้ได้ว่า “ผิดกฎอะไร” (rule_id + severity + delta)
  • เปิดได้ว่า “อ้างอิงเอกสารหน้าไหน” (doc/page/snippet)
  • ตอบได้ว่า “ใครตัดสินใจ เพราะอะไร” (mandatory reason)

Node Map

TH8 เข้าไปช่วยตรงไหน (แบบชี้ node ต่อ node)

สรุปให้ผู้บริหารเห็นว่า “ระบบเดิมทำอะไร” และ “TH8 เติมอะไร” โดยไม่บังคับให้เปลี่ยน SAP/Workflow

SAP = System of Record TH8 = System of Decision
Node (งาน) ระบบเดิมทำได้ ช่องว่างที่เจอจริง TH8 เติมให้ ผลลัพธ์ (ดีขึ้น)
PO/PR Entry บันทึกธุรกรรม, คุม master data ยังไม่ “เทียบสัญญา” ให้เป็นเหตุผลได้ อ่านข้อมูลเพื่อทำ policy check แยก “เคสปกติ” ออกจาก “เคสเสี่ยง” ตั้งแต่ต้น
Document/OCR แปลงภาพ/ไฟล์เป็นข้อความ ข้อความไม่ผูกกับจุดตัดสินใจ, หา page/snippet ยาก Evidence (doc/page/snippet) เปิดหลักฐานได้ทันที ลดเวลาไล่เอกสาร
Approval Workflow ลำดับอนุมัติ, routing ตามวงเงิน ไม่บังคับเหตุผล, ไม่ชี้ delta, ไม่รวมหลักฐาน Decision card + mandatory reason ตัดสินใจเร็วขึ้น + เหตุผลเป็นมาตรฐาน
Policy/Controls ตั้ง rule ได้บางส่วน (config) rule ไม่ผูกหลักฐาน, pending เกิดแบบอธิบายยาก policy → pending reason → decision → audit Pending “เกิดจริง” และ audit ได้
Dashboard/Audit รายงานภาพรวม, audit log เบื้องต้น ไล่เหตุผลรายเคสลำบาก, export ไม่ครบ immutable event + export CSV/JSON ตรวจย้อนหลังได้เร็ว, ลด effort audit

Outcomes

สุดท้าย “ดีขึ้นยังไง” (พูดกับผู้บริหารได้แบบไม่ต้องเชื่อ AI)

1) เวลาอนุมัติลดลง

CFO เห็น delta+หลักฐานในจอเดียว

UX change

จาก “ไล่เอกสาร” → “กดตัดสินใจบน evidence”

2) ลดความเสี่ยงเชิงควบคุม

เหตุผลบังคับ + ผูกกับ rule + policy version

Control change

จาก “free-text” → “มาตรฐานตรวจสอบได้”

3) KPI ชี้ผลได้

Amount at risk / prevented leakage คำนวณจากเหตุการณ์จริง

Reporting change

จาก “รายงานหลังเกิดเหตุ” → “เห็นความเสี่ยงขณะตัดสินใจ”

KPI Examples

ตัวอย่าง KPI ที่ “derive จาก data”

Amount at Risk

Σ pending PO amount

สะท้อน “มูลค่าที่ต้องให้ผู้บริหารโฟกัส”

Prevented Leakage

Σ corrected/rejected delta

ชี้ “มูลค่าที่กันไว้ได้” จากการตัดสินใจ

Decision SLA

within_sla / total_pending

วัดความเร็วการตัดสินใจแบบจับต้องได้

Board-ready Talk Track

ระบบเดิมดีอยู่แล้ว

SAP/Workflow เก่งเรื่อง transaction + routing

แต่จุดเจ็บคือ “การตัดสินใจ”

เทียบสัญญา + เหตุผล + หลักฐาน ยังต้องพึ่งคนไล่เอง

TH8 ทำให้ตัดสินใจเป็นมาตรฐาน

Pending ที่ควรดูจริง + evidence link + reason บังคับ + audit export
