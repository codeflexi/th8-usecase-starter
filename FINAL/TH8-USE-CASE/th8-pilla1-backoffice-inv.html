คุยกับ Inventory Agent

“ช่วยสรุป Top 10 SKU ที่เสี่ยงขาดสต็อกใน 7 วันข้างหน้า โดยแยกตามสาขา และระบุจำนวนที่แนะนำให้สั่งเพิ่มแต่ละที่”

สรุปเบื้องต้นจากข้อมูล 30 วันที่ผ่านมา:

  • พบ SKU เสี่ยงขาดสต็อกใน 7 วันรวม 18 รายการ
  • กระจายอยู่ใน 3 สาขาหลัก: เชียงใหม่, ลำพูน, ลำปาง
  • Draft PO เตรียมไว้แล้ว 3 ฉบับ แยกตามผู้ขายหลัก

คุณสามารถเปิดดู / แก้ไข Draft PO ก่อนกดอนุมัติและส่งเข้า ERP ได้

DT

Inventory Data Set Overview

สำหรับ Back Office / Data / IT

Data refresh: ทุก 15 นาที

inventory_snapshot

Primary

ภาพรวมสต็อกปัจจุบันราย SKU x สาขา · มี field สำหรับ stock_qty, safety_stock, on_order, reserved.

Source: WMS + ERP (Stock module)

sales_history_daily

Demand Pattern

ยอดขายรายวัน 12–18 เดือนย้อนหลัง แยกตาม SKU x สาขา · มี rolling avg, seasonality flag.

Source: POS / ERP (Sales module)

purchase_orders / inbound_shipment

Supply Side

ข้อมูล PO ที่เปิดแล้ว / กำลังส่งของ · ใช้คำนวณ inbound_qty, lead time จริง, GR delay.

Source: ERP (Purchasing), WMS (GR/Putaway)

config_inventory_policy

Policy & Rule

เก็บ Policy ต่อ SKU/กลุ่ม เช่น min/max days of cover, safety_factor, channel priority.

ใช้กำหนด Guardrail ให้ Agent ไม่เสนอคำแนะนำเกินขอบเขตที่ธุรกิจต้องการ

IN

Integration Console · การเชื่อมต่อระบบ

เห็นสถานะการเชื่อม ERP, WMS, POS, eCommerce, Sheet

ERP – Stock & Purchasing

ใช้ดึงสต็อกกลาง, PO, Supplier, Cost และส่ง Draft PO ที่ Agent แนะนำกลับเข้าไป.

Status: Connected · API (REST) · Refresh ทุก 15 นาที

WMS – คลังสินค้า

ใช้ข้อมูล GR, Putaway, Picking, Transfer เพื่ออัปเดต available stock และ lead time จริง.

Status: Connected · DB View / ETL ทุก 30 นาที

POS & eCommerce

ใช้สร้างยอดขายรายวัน แยกตามช่องทาง เพื่อให้ Agent เห็น pattern Demand จริง.

Status: Partial · POS API OK · eCommerce CSV Upload (กำลังวาง API)

Google Sheet / Manual Upload

ใช้เก็บ Policy, Temporary override และรายการสินค้าทดลองที่ยังไม่มีใน ERP/WMS.

Status: Connected · ใช้สำหรับ config และ one-off data

Flow การทำงานของ Inventory Agent (สรุปสั้น ๆ)

  1. ดึงข้อมูล stock, sales, PO, policy จาก data set ด้านบน
  2. คำนวณ demand forecast และ days of cover ต่อ SKU x สาขา
  3. ตรวจ rule / policy จาก config_inventory_policy
  4. สร้างคำแนะนำ Replenishment และ Draft PO (ยังไม่ post จริง)
  5. รออนุมัติจากคนก่อนยิงเข้า ERP · ทุก action เก็บ log เพื่อใช้วิเคราะห์ต่อ
