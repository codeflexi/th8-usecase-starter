CFO — ROI / Payback / Cost Control

ตอบให้ชัด: ลงทุนเท่าไร ได้คืนจากไหน คุมต้นทุน AI ยังไง

ROI มาจากอะไรบ้าง? (ไม่ใช่แค่ “เพิ่มรายได้”) ⌄

Complaint Agent: ลด cost handling + ลด escalation ที่ไม่จำเป็น + ลดความเสี่ยง brand จากการตอบช้า/ตอบผิด

FP&A Agent: ลดวันคนทำงาน + ลด rework/meeting + ปรับแผน/forecast ได้เร็วขึ้น

หมายเหตุ: brand upside เป็น “option value” — CFO มักใส่เป็น conservative (ไม่นับ) ก็ยังคุ้ม

คุมต้นทุน AI อย่างไร? จะไม่บานปลายเหรอ? ⌄

TH8 ใช้แนวทาง SLM-first: งาน routine ให้ SLM ทำก่อน (เร็ว/ถูก) แล้วค่อย escalate ไป LLM เมื่อจำเป็น

  • มี routing rules แยก task type
  • มี confidence gate + schema validation ลด retry ที่ไม่จำเป็น
  • มี metrics/cost tracking ราย task / ราย channel
Payback ควรคาดหวังกี่เดือน? ⌄

โดยทั่วไปสำหรับ 2 use case นี้ ถ้าใช้ตัวเลข conservative: ~9–12 เดือน

ตัวเลขจริงขึ้นอยู่กับ volume complaint, cost per case และ effort ต่อเดือนของทีม FP&A

ถ้าไม่ทำอะไรเลย จะเกิดอะไรใน 6–12 เดือน? ⌄
  • Complaint: ตอบช้า/ไม่สม่ำเสมอ → escalation มาก → brand risk สูงขึ้น
  • FP&A: variance cycle ยังช้า → forecast ปรับไม่ทัน → margin management ยาก

IT — Architecture / Integration / Security

ตอบให้ชัด: ต่อระบบอย่างไร คุมสิทธิ์อย่างไร ตรวจสอบอย่างไร

ต้อง integrate อะไรบ้าง? เริ่มแบบเบา ๆ ได้ไหม? ⌄

เริ่มแบบเบาได้: ใช้ export รายวัน/รายสัปดาห์ (CSV/DB view) ก่อน แล้วค่อยขยับเป็น API/stream

  • Complaint: contact center/CRM/LINE OA export + policy docs
  • FP&A: plan/actual table + promo calendar + driver tables
Tool Permission คืออะไร? ทำไมต้องมี? ⌄

TH8 ใช้ allow-list ต่อ task type ว่า agent ใช้ tool อะไรได้บ้าง

ตัวอย่าง: complaint_pipeline อนุญาต “sql lookup” ได้ แต่ห้าม “write back” โดยไม่ผ่าน approval

Observability มีอะไรบ้าง? (debug / audit / cost) ⌄
  • Trace ID ต่อ request
  • Latency ต่อ node (SLM/LLM/tool)
  • Retry reason (schema-invalid vs transient)
  • Cost/usage summary ต่อเดือน

เริ่มจาก jsonl logs แล้วค่อย push ไป OTEL/Prometheus ได้

Deploy แบบไหน? Cloud / VPC / On-prem? ⌄

รองรับได้หลายแบบตาม policy: Cloud, VPC, หรือ on-prem

Enterprise มักเริ่มใน VPC + private networking + restricted egress

Legal — PDPA / PII / Audit Trail

ตอบให้ชัด: ข้อมูลส่วนบุคคลไปไหน เก็บอะไร บันทึกอย่างไร

TH8 จัดการ PII อย่างไร? ⌄

ก่อนส่งเข้าโมเดล TH8 ทำ PII redaction (mask) และเก็บ audit trail ต่อ field

  • Mask: เบอร์โทร, อีเมล, ชื่อ/นามสกุล (ตาม rule)
  • Audit: field ไหนถูก mask, ด้วย rule อะไร, เวลาไหน
  • Default: ไม่ส่ง PII ออกนอก boundary ที่ลูกค้ากำหนด
Data Retention / Deletion ทำได้ไหม? ⌄

ทำได้ โดยแยกข้อมูลเป็น 2 ส่วน:

  • Business logs (trace/decision) เก็บตาม policy
  • PII source data เก็บตามระบบต้นทาง และลบตาม retention rule

สิ่งที่ TH8 ต้องเก็บเพื่อ audit คือ “เหตุผล/หลักฐาน” ไม่จำเป็นต้องเก็บ PII ดิบ

ถ้าโมเดลตอบผิด ใครรับผิดชอบ? ⌄

TH8 วางเป็น Decision Support + Controlled Automation ไม่ใช่ autonomous สุ่มทำเอง

  • มี confidence gate: ต่ำกว่ากำหนด → escalate / human approval
  • มี policy guardrails: ห้าม action บางประเภทโดยอัตโนมัติ
  • มี audit trail: trace ได้ว่าใช้ context อะไร ตัดสินใจเพราะอะไร
Model training ใช้ข้อมูลลูกค้าไหม? ⌄

ค่าเริ่มต้น: ไม่ใช้ข้อมูลลูกค้าไป train model กลาง

ถ้าจะทำ fine-tune/SLM specialist: ใช้เฉพาะ data ที่ anonymized และอยู่ใน environment ของลูกค้า

Checklist ที่ควรได้ก่อน Go-live

CFO
  • ROI assumptions agreed
  • Cost cap / routing policy
  • Monthly outcome report
IT
  • RBAC / network boundary
  • Logging + monitoring
  • Rollback plan
Legal
  • PDPA mapping
  • Retention policy
  • Audit trail scope
