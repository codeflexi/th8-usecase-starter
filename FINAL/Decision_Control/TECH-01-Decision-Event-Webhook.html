End-to-End Flow (Workflow + Data Flow)

แสดง 2 เส้นทางพร้อมกัน: Workflow path (การอนุมัติในระบบเดิม) และ Decision/Audit path (บริบท+policy+หลักฐานใน TH8)

Conceptual

A) Workflow path (ระบบเดิม)

Approve in existing UX
  1. 1

    Create / update transaction

    PO/GR/Invoice เกิดใน SAP หรือ procurement system

  2. 2

    Workflow routes the task

    ระบบ workflow ส่ง task ตาม approval chain

  3. 3

    Approver clicks Approve/Reject

    ผู้บริหาร “กดอนุมัติ” ในระบบเดิมที่คุ้นเคย

  4. 4

    ERP executes & updates status

    SAP เปลี่ยนสถานะและทำธุรกรรมต่อ

เป้าหมาย UX

ไม่บังคับผู้บริหารย้ายที่กดอนุมัติ — แค่ “เพิ่มข้อมูลตัดสินใจ” ให้ครบก่อนกด

B) Decision & Audit path (TH8)

Decision Control Layer
  1. 1

    Ingest signals & normalize case

    ดึง transaction + workflow status + timestamps → สร้าง “Case” แบบเดียว

  2. 2

    Evaluate policy (versioned)

    ตรวจ authority / contract clause / SLA / override rules ตาม policy version

  3. 3

    Retrieve evidence (RAG as evidence)

    ดึงเอกสารที่ “อนุญาตเท่านั้น” พร้อม citation + link

  4. 4

    Publish decision brief

    สรุปสิ่งที่ผู้บริหารต้องรู้: “เสี่ยงอะไร / เงินเท่าไร / ต้องใครอนุมัติ / อ้างอิงอะไร” แล้วแนบเข้า UX เดิม (overlay/link)

  5. 5

    Record timeline across systems

    เก็บ event จาก SAP/Workflow/TH8 รวมเป็น timeline เดียว (source-tagged)

  6. 6

    Audit pack ready

    สร้างรายงานสำหรับ audit: decision log + policy version + evidence list

Pattern 1: Inline overlay

แสดง Decision Brief “ข้างปุ่มอนุมัติ” ในระบบเดิม

  • • ไม่เปลี่ยน UX ของผู้บริหาร
  • • ลดการสลับหน้าจอ
  • • เหมาะกับ phase 1

Pattern 2: Executive inbox

TH8 คัดเฉพาะเคสสำคัญ → ส่งให้ผู้บริหาร

  • • ลด noise จาก task list
  • • โฟกัส “impact สูง”
  • • เหมาะกับ mobile/board

Pattern 3: Deep integration

SSO + embedded UI + context injection

  • • เหมาะเมื่อ trust สูง
  • • ต้องมี IT/SI รองรับ
  • • ทำหลังพิสูจน์ ROI

What TH8 must prepare (ขายได้จริง)

Integration Package ที่ TH8 ควรเตรียม “ให้ครบ”

เพื่อไม่โยนภาระให้ลูกค้าออกแบบเอง และลดความเสี่ยงการ implement ในองค์กร

Enterprise Readiness

✅ Event schema มาตรฐาน

Mandatory

TH8 ต้องกำหนดรูปแบบ event ให้ชัด เพื่อให้ทุกระบบส่งข้อมูลได้เหมือนกัน (ไม่ให้ลูกค้าออกแบบเอง)

ตัวอย่าง fields ที่ควรมี

  • • event_type (APPROVED / REJECTED / TASK_CREATED …)
  • • source_system (SAP / WORKFLOW / TH8)
  • • business_ref (PO id / invoice id / case id)
  • • actor (user/system) + timestamp
  • • policy_version + evidence_links (ถ้ามี)
JSON schema Validation Versioning

✅ Mapping template ต่อ SAP / Workflow

Implementation kit

ลูกค้าไม่ควรเริ่มจากศูนย์ — TH8 ควรมี template mapping “เหตุการณ์ยอดนิยม” และ field mapping ที่ใช้จริง

SAP mapping

PO status · release strategy · approver id · timestamps

Workflow mapping

task created · assigned_to · completed · SLA signals

สิ่งที่ควรมี

  • • Field mapping table (source → TH8 event)
  • • Sample payload (copy-paste)
  • • Correlation rules (ใช้ PO/Invoice/Case key ไหนเป็นหลัก)

✅ iPaaS examples

Adoption booster

ลดแรง IT: ให้ตัวอย่าง workflow สำเร็จรูปสำหรับ iPaaS ยอดนิยมในองค์กร (เลือกให้เข้ากับตลาดไทยได้)

Webhook → TH8 events

เมื่อ task complete ส่ง event เข้า TH8

Scheduled sync

ดึง status ทุก 5–15 นาที (fallback ของ real-time)

ควรเตรียม

  • • n8n / Power Automate / Make / Mule / Boomi recipes
  • • ตัวอย่างแปลง payload + retry policy
  • • Error handling + dead-letter pattern

✅ Fallback mode (manual / snapshot)

Phase 0–1

สำหรับลูกค้าที่ “แตะระบบยาก” TH8 ต้องยังใช้ได้ โดยระบุชัดว่า event มาจากอะไร เพื่อไม่ทำให้ audit สับสน

Manual event

เจ้าหน้าที่กด “record event” พร้อมแนบหลักฐาน

Snapshot ingestion

รับ email/CSV/report แล้วสร้าง timeline แบบติดป้าย

ต้องบังคับใน UI

  • • แสดง source_system = MANUAL / SNAPSHOT ชัดเจน
  • • Require attachment (หลักฐาน) สำหรับ event สำคัญ
  • • ระบุ level: management audit vs system audit

อื่น ๆ ที่จำเป็น (เพื่อ enterprise จริง)

TH8 ต้องเตรียมให้ครบเพื่อให้ลูกค้ากล้าขึ้นระบบ

Non-negotiable

SSO + RBAC

สิทธิ์ตามบทบาท (CFO/COO/Audit/Ops) + separation of duties

Event validation + idempotency

กัน event ซ้ำ, ตรวจ schema, เก็บ error อย่างเป็นระบบ

Correlation & reconciliation

ผูกเหตุการณ์ข้ามระบบด้วย business key และมีหน้าตรวจ “ข้อมูลไม่ตรง”

Immutable audit logs

ป้องกันการแก้ย้อนหลัง + export audit pack ได้

Evidence governance

กำหนด allowed sources, versioning, review/publish ก่อนใช้ใน RAG

Operational monitoring

health check integrations, latency, backlog, alerting

Key message (ใช้ปิดการขาย)

TH8 ไม่ได้มาแทน workflow ของ SAP แต่ทำให้ทุกการอนุมัติ “มีข้อมูลครบก่อนกด” และทุกการตัดสินใจ “ตรวจสอบย้อนหลังได้” แม้ action จะเกิดในระบบเดิม

Tip: เวลานำเสนอ ให้ชี้ 2 เส้นทางคู่กัน: “Workflow path” (ของเดิม) + “Decision/Audit path” (TH8) แล้วปิดด้วย Integration Package ที่ TH8 เตรียมให้ครบ
