Use case overview

จัดการเคสร้องเรียน/รีวิว/ข้อความ LINE OA แบบ end-to-end: รับเรื่อง → จัดหมวด/ความรุนแรง → ตรวจข้อมูลออเดอร์/ล็อต → สร้างคำตอบตาม policy → ปิดเคส พร้อม audit ต่อ field

What changes
งานที่ “ทำซ้ำทุกวัน” กลายเป็น pipeline ที่คุมได้
  • แยก deterministic vs probabilistic ก่อนเรียก model
  • ใช้ SLM specialist ทำงานย่อย → เร็ว/ถูก/คุม cost
  • LLM ใช้เฉพาะงานต้อง reasoning จริง
Guardrails
ปลอดภัย + ตรวจย้อนหลังได้
  • PII redaction + field-level audit trail
  • confidence gate + schema validation
  • targeted retry (เฉพาะ schema-invalid) + escalate
Recommended actions (auto)
สิ่งที่ระบบจะส่งให้ทีมทำต่อแบบ “ทำงานได้ทันที”
1) ตอบลูกค้าด้วย template ที่ถูก policy พร้อมขอหลักฐาน (รูป/LOT/วันหมดอายุ)
2) แนบหลักฐานจาก order_lookup (ออเดอร์/ล็อต/สถานะจัดส่ง) เพื่อกันข้อโต้แย้ง
3) ถ้า “quality complaint” เกิน threshold ต่อ batch/สัปดาห์ → สร้าง QA sampling ticket อัตโนมัติ

Context Tables (Schema) + Example Data

ชุด schema ขั้นต่ำที่ทำให้ agent “เห็นข้อมูลถูกชั้น” และใช้งานจริงได้

L1 · complaint_events
Real-time
ข้อความ/โพสต์/เคสรายเหตุการณ์ (ดิบ) — ต้อง redaction ก่อนใช้กับ AI
Schema
case_id (string)
channel (LINE|email|social)
received_at (timestamp)
message_text (string)
customer_ref (string, optional)
order_ref (string, optional)
Example
{ "case_id":"CC-2025-12-0231","channel":"LINE","received_at":"2025-12-20T03:11:00Z","message_text":"เมื่อวานซื้อโอวัลตินแล้วชงเป็นก้อน โทร 08x-xxx-xxxx","order_ref":"ORD-77821" }
L2 · policy_templates
Change-based
นโยบาย/เทมเพลตคำตอบ (truth) + version เพื่อคุม risk
Schema
policy_id (string)
policy_type (refund|quality|delivery)
allowed_actions (array)
response_template (string)
version (string)
effective_date (date)
Example
{ "policy_id":"POL-REFUND-01","policy_type":"refund","allowed_actions":["ASK_PHOTO","OFFER_REPLACEMENT","ESCALATE_MANAGER"],"response_template":"ขออภัยค่ะ รบกวนส่งรูปสินค้า/วันหมดอายุ...","version":"v2.1","effective_date":"2025-08-01" }
L3 · customer_case_history
Daily / Weekly
สรุปประวัติเคสเดิม (ไม่ส่ง raw log เข้า LLM) เพื่อช่วยตัดสินใจ escalation
Schema
customer_ref (string)
period (YYYY-MM)
case_count (int)
repeat_issue (string)
last_resolution (string)
Example
{ "customer_ref":"CUST-10293","period":"2025-11","case_count":2,"repeat_issue":"clumping","last_resolution":"replacement_sent" }
L4 · order_lookup
Near real-time
ข้อมูลออเดอร์/ล็อต/ขนส่ง เพื่อยืนยันข้อเท็จจริง (deterministic)
Schema
order_ref (string)
product_code (string)
batch_id (string)
ship_status (string)
delivery_date (date)
channel (string)
Example
{ "order_ref":"ORD-77821","product_code":"OVALTINE-ORIGINAL","batch_id":"OVT-2512-05","ship_status":"DELIVERED","delivery_date":"2025-12-19","channel":"eCommerce" }
Key rule
ประหยัด + คุมได้
อย่า dump ข้อมูลดิบทั้งหมดเข้าโมเดล
สร้าง “evidence pack” ที่ clean + relevant + traceable ก่อน แล้วค่อยให้ agent ตัดสินใจ/สร้าง output (ลด token, ลด hallucination, audit ได้)
